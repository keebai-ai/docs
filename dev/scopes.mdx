Cada Personal Access Token (PAT) tiene un conjunto de **scopes**: el listado de acciones que ese token puede ejercer contra la API pública. Los scopes se eligen al crear el token y no se modifican después — para cambiarlos hay que generar un token nuevo y revocar el viejo.

## Cómo se validan

En cada request, el guard de autenticación carga el token y deja sus scopes en el contexto. Cada endpoint declara qué scope necesita con `@RequireScopes('messages:send')`. Si el scope falta, la respuesta es `403 Forbidden` con `code: INSUFFICIENT_SCOPE`.

```http
HTTP/1.1 403 Forbidden
Content-Type: application/json
```

```json
{
  "error": {
    "code": "INSUFFICIENT_SCOPE",
    "message": "Forbidden",
    "details": {
      "required": ["messages:bulk"],
      "missing": ["messages:bulk"]
    }
  }
}
```

A diferencia de los errores de autenticación (que son intencionalmente genéricos), los errores de autorización indican qué scope falta — porque ya autenticaste correctamente, solo te falta permiso.

## Catálogo de scopes

El catálogo vive en la colección `api_scopes` de MongoDB. El portal lo consume vía `GET /api/v1/api-scopes` y arma dinámicamente los checkboxes del modal de creación, así que agregar un scope nuevo no requiere deploy del frontend.

| Scope | Grupo | Acción |
|---|---|---|
| `messages:send` | messages | Envío de un mensaje individual (texto o template) a un destinatario. |
| `messages:bulk` | messages | Envío masivo de un template a múltiples destinatarios y consulta del estado de la campaña. |
| `templates:read` | templates | Lectura del catálogo de templates aprobados de la company. |
| `templates:create` | templates | Crear templates de WhatsApp y enviarlos a Meta para aprobación. |
| `templates:update` | templates | Editar un template existente (componentes, categoría, idioma, status). |
| `channels:read` | channels | Listar canales (`/v1/channels`) y números WhatsApp (`/v1/whatsapp/numbers`). |
| `channels:connect` | channels | Iniciar y pollear sesiones de [conexión de WhatsApp Business](/dev/endpoints/channels-whatsapp-connect) desde el CLI o un backend propio. |
| `webhooks:read` | webhooks | Listar subscripciones de [webhooks salientes](/dev/webhooks/overview) y consultar deliveries. |
| `webhooks:manage` | webhooks | Crear, actualizar, rotar secret, eliminar subscripciones y disparar eventos sintéticos. |
| `knowledge:read` | knowledge | Lectura del [árbol de carpetas y documentos](/dev/endpoints/knowledge-tree) de la base de conocimientos. |
| `knowledge:write` | knowledge | Crear [carpetas](/dev/endpoints/knowledge-folder-create) y [documentos](/dev/endpoints/knowledge-document-create) (markdown → BlockNote). |
| `knowledge:delete` | knowledge | [Eliminar nodos](/dev/endpoints/knowledge-node-delete) (carpetas y documentos) de la base de conocimientos. |
| `knowledge:query` | knowledge | Consultar la base de conocimientos con [búsqueda full-text, vectorial o híbrida](/dev/endpoints/knowledge-search). |
| `knowledge:assign` | knowledge | [Asignar nodos](/dev/endpoints/knowledge-assign) de la base de conocimientos a los workers de un asistente. |
| `assistants:read` | assistants | [Listar y buscar asistentes](/dev/endpoints/assistants-list) activos de la company. |

<Tip>
  La lista que aparece en el modal de **Crear API Key** se carga en vivo desde el backend. Si un scope no está, es porque está marcado como `is_active: false` en la base.
</Tip>

## Independencia de los permisos del rol

Los scopes son **independientes** de los permisos del rol del usuario que crea el token. La validación de qué scopes podés asignar ocurre **al momento de crear** la key (el backend chequea que cada scope exista y esté activo en `api_scopes`), pero una vez emitido, el token funciona con los scopes que tiene grabados sin importar cambios posteriores en el rol del usuario.

Esto significa:

1. **Para crear un token necesitás el permiso `developer.manage_tokens`** en tu rol del portal. Sin ese permiso ni siquiera ves la pantalla de API Keys.
2. **Una vez creado, los scopes del token son fijos**. Si después te quitan permisos en el portal, el token sigue funcionando con sus scopes originales. Para revocar el acceso, hay que revocar la key explícitamente.
3. **Para usar scopes nuevos hay que crear un token nuevo**. No se pueden agregar scopes a una key existente.

<Warning>
  Como los scopes sobreviven a cambios en el rol, **revisá periódicamente la tabla de API Keys** y revocá los tokens de personas que ya no deberían tener acceso. La columna *Último uso* ayuda a detectar tokens olvidados.
</Warning>

## Principio de menor privilegio

Asigná solo los scopes que el token realmente necesita. La separación de `messages:send` vs `messages:bulk` está pensada exactamente para esto: el bulk es más sensible (costo de Meta, reputación del número emisor, posibles abusos) y conviene aislarlo en una key controlada por la herramienta de marketing, no en la key del backend transaccional.

## Cambiar los scopes de un token

No se pueden modificar los scopes de un token existente. El flujo es:

<Steps>
  <Step title="Crear un token nuevo con los scopes correctos">
    Desde [API Keys](https://app.keebai.com/system-settings/api-keys), clic en **Crear API Key** y elegí los nuevos scopes.
  </Step>
  <Step title="Actualizar tu integración">
    Reemplazá el token viejo por el nuevo en tu gestor de secretos.
  </Step>
  <Step title="Revocar el viejo">
    Una vez confirmado que la integración funciona con el token nuevo, revocá el viejo.
  </Step>
</Steps>

Este flujo te permite hacer la rotación sin downtime de la integración.
